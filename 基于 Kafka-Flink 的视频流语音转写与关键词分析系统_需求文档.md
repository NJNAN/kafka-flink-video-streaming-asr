**项目名称**：StreamSense — 多模态视频流实时理解系统

**版本**：v1.0
**日期**：2026-04-24
**类型**：工业级实时数据处理 + AI 推理系统
**保密级别**：内部技术方案

---

## 一、项目背景与目标

### 1.1 业务场景

实时视频流（直播、监控、录播）在进入下游AI分析前，需要完成两项基础工作：把视频里的语音转成文字（ASR），以及对这段内容快速提取关键词做索引/告警。现有方案有两种常见做法：

- **方案A**：用ffmpeg提取音频 → 调用云端ASR API → 人工处理结果 → 输出
- **方案B**：直接调用视频理解大模型对每一帧做分析

方案A延迟低但依赖外部服务，无法私有化部署；方案B精度高但成本极高，且大模型不适合做纯粹的语音转写任务。

本项目目标是构建一套**端到端私有化的实时视频流理解系统**：从视频输入到字幕+关键词输出，全链路可本地部署、可水平扩展、成本可控。

### 1.2 系统目标

| 指标 | 目标值 |
|------|--------|
| 端到端延迟（从视频帧到字幕输出）| < 5秒 |
| 支持并发视频流数量 | ≥ 8路（可扩展）|
| 关键词召回率 | ≥ 70%（对比人工标注）|
| 系统可用性 | 99%（24小时连续运行）|
| 部署方式 | Docker Compose，一键启动 |

## 二、系统架构

### 2.1 整体数据流

```
[视频源] 
  │
  │  RTSP / HTTP-FLV / 录播文件
  ▼
[Kafka 消息队列]
  │  Topic: raw-video-stream, transcription-result, keyword-event
  ▼
[Flink 流处理引擎]
  │  算子: 视频切片 → 音频提取 → 语音分段 → 推理调度 → 结果组装
  ▼
[Whisper ASR 服务]
  │  本地部署（GPU/CPU），HTTP接口，批处理+流式双模式
  ▼
[KeyBERT 关键词服务]
  │  接收字幕文本，输出Top-N关键词 + 置信度
  ▼
[Redis 状态缓存]
  │  存储最近字幕历史（用于语义比对）、关键词事件
  ▼
[Dashboard / API Gateway]
      实时字幕流 + 关键词事件时间线 + 系统监控
```

### 2.2 核心设计决策

**决策一：Kafka 作为数据中枢**
- 解耦视频采集与AI推理两个处理速度不同的环节
- 支持多消费者并行订阅同一视频流（字幕、关键词分别消费）
- Kafka分区策略：按视频源ID哈希分区，保证同一视频的帧有序

**决策二：Flink 作为流处理编排层**
- 使用事件时间（Event Time）而非处理时间，保证跨流同步
- Flink状态后端使用RocksDB，支持故障恢复后的状态恢复
- 精确一次语义（Exactly-Once）保证关键词事件不重复、不丢失

**决策三：Whisper 本地部署**
- 不依赖外部云ASR服务，支持私有化部署
- 模型可选：base（CPU可用）或small（GPU优先）
- 支持流式分片推理：音频边接收边转写，无需等待完整音频

**决策四：关键词与字幕同一管道输出**
- KeyBERT直接消费Whisper输出的字幕流，无需独立数据通道
- 语义跳转检测：当前后两条字幕相似度骤降时，触发关键词事件上报

| REQ-003 | 支持多路视频流并发接入（≥8路）| 必须 |
| REQ-004 | 每路视频流独立配置：分辨率、帧率、音频编码 | 应该 |

| 需求ID | 描述 | 优先级 |
|--------|------|:------:|

| REQ-007 | 支持VAD（语音活动检测），跳过无语音片段 | 应该 |
| REQ-008 | 音频质量降级时（信噪比低）自动降级处理策略 | 可以 |

### 3.3 ASR推理

| 需求ID | 描述 | 优先级 |
|--------|------|:------:|
| REQ-009 | Whisper base/small模型本地部署，HTTP接口 | 必须 |

| REQ-012 | 支持中文普通话、英语双语识别 | 应该 |
| REQ-013 | 无GPU环境下CPU推理延迟 < 10秒/3秒音频 | 应该 |

### 3.4 关键词事件检测

| 需求ID | 描述 | 优先级 |
|--------|------|:------:|

| REQ-016 | 关键词事件包含：关键词、触发时间戳、置信度、来源字幕 | 必须 |
| REQ-017 | 支持关键词自定义词表（行业术语优先抽取）| 应该 |

| REQ-020 | 历史字幕和事件持久化至文件存储（用于回溯）| 应该 |

### 3.6 监控与可观测性

## 四、非功能需求

### 4.1 性能

| 指标 | 要求 |
|------|------|
| 端到端延迟（P95）| < 5秒（3秒音频片段）|
| 端到端延迟（P99）| < 8秒 |
| 吞吐量 | 单GPU机器 ≥ 4路并发（3秒/路）|
| 内存占用 | Flink JobManager < 2GB；Worker < 4GB；Whisper < 8GB |

### 4.2 可用性

- 故障恢复后自动重新消费Kafka未确认消息
- Flink checkpoint间隔 ≤ 30秒
- 单组件故障不影响其他视频流继续处理

### 4.3 可维护性

- 所有组件通过Docker Compose管理，无特殊环境依赖
- 配置文件外置，支持不修改镜像即可调整参数
- 日志格式统一（JSON），支持日志聚合

## 七、验收标准

### 7.1 功能验收

- [ ] 单路视频流接入，从启动到首条字幕输出 < 10秒
- [ ] 连续运行30分钟，无崩溃、无内存泄漏
- [ ] 关键词事件在语义跳转后2秒内出现在Dashboard

### 7.2 性能验收

- [ ] 端到端P95延迟 < 5秒（单路，有GPU）
- [ ] 支持4路以上并发（CPU模式）

## 八、风险与边界

| 风险 | 缓解措施 |
|------|---------|
| Whisper CPU推理速度不足 | 提供base模型+INT8量化方案；预留GPU机器选项 |
| Flink状态过大导致OOM | 配置状态TTL；定期checkpoint清理 |
| 音频VAD误判导致漏转写 | 提供全量音频处理模式（无VAD）|
| 关键词召回率不达标 | 引入多模型融合（KeyBERT + 词频统计）|
