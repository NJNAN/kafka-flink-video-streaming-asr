## 3. 增强一：字幕质量评测闭环

### 3.1 为什么这个最值得做

现在项目能生成字幕，也能统计覆盖缺口和处理耗时，但“字幕准不准”主要还是人工看。答辩时如果老师问“你这个转写效果怎么评价”，只展示几句字幕会显得主观。

建议增加一个评测脚本，支持用人工整理的参考文本或参考字幕作为标准答案，计算：

- CER，Character Error Rate，中文更适合用字符错误率。
- WER，Word Error Rate，如果文本中有英文或空格分词，可以辅助统计。
- 关键词命中率，例如 Kafka、Flink、Whisper、ASR、Redis 等词有没有识别出来。
- 字幕覆盖率，复用现有 `generate_video_subtitles.py` 的补漏报告字段。
- 低置信度片段数量，用 ASR 返回的 `avg_logprob`、`no_speech_prob` 等字段辅助判断。
- 人工修改前后差异，如果桌面端保存了编辑后的字幕，可以比较修改量。

这会让项目从“我觉得效果还可以”变成“我有指标证明效果”。

```python
async def collect_metrics_history():
```

## 6. 增强四：失败重试和死信记录

### 6.1 为什么值得做

当前 Flink 调用 ASR 时，如果 ASR 超时或报错，会返回 error 状态，API 也会记录失败片段。这个已经能说明错误被发现了。

但工程完整性还可以继续提高：失败片段应该先重试，重试仍失败后再进入死信记录。这样能体现“流处理容错设计”。

### 6.2 建议修改文件

```text
docker-compose.yml
flink/transcription_job.py
services/api/app.py
docs/system_metrics.md
```

### 6.3 设计思路

建议增加一个死信 topic：

```text
transcription-failed
```

处理流程：

```text
audio-segment
-> Flink 调用 ASR
-> 成功：写 transcription-result
-> 失败：重试 2 次
-> 仍失败：写 transcription-result(status=error)，同时写 transcription-failed
-> API 记录 failed_segments.jsonl
```

如果不想增加第二个 Flink Sink，可以先只把错误详细信息放到 `transcription-result`，API 继续写 `failed_segments.jsonl`。这属于简化版。

### 6.4 具体实现步骤

第一步，在 `.env.example` 增加：

```text

ASR_RETRY_TIMES=2
ASR_RETRY_BACKOFF_MS=800
```

第二步，在 `docker-compose.yml` 的 `topic-init` 增加 topic 创建：

```bash

```

同时给 `topic-init` 和 `flink-job-submitter` 加环境变量：

```yaml

ASR_RETRY_TIMES: ${ASR_RETRY_TIMES:-2}

```

第三步，在 `flink/transcription_job.py` 增加重试函数。

```python
def call_asr_with_retry(request_body):
    retry_times = int(os.getenv("ASR_RETRY_TIMES", "2"))
    backoff_ms = int(os.getenv("ASR_RETRY_BACKOFF_MS", "800"))
    last_error = ""

    for attempt in range(retry_times + 1):
        try:
            response = requests.post(f"{ASR_URL}/transcribe", json=request_body, timeout=ASR_TIMEOUT_SECONDS)
            response.raise_for_status()
            result = response.json()
            result["retry_count"] = attempt
            return result, ""
        except Exception as exc:
            last_error = str(exc)
            if attempt < retry_times:
                time.sleep(backoff_ms / 1000)

    return {
        **request_body,
        "text": "",
        "language": "",
        "inference_time_ms": 0,
        "retry_count": retry_times
    }, last_error
```

第四步，在 `transcribe_segment()` 里替换原本直接 `requests.post()` 的逻辑。

成功时：

```python
status = result.get("status", "ok")
```

失败时：

```python
status = "error"
error = last_error
```

第五步，在 API 指标里增加：

```text

retry_count_average
```

当前 `recent_errors` 已经存在，可以补一个 retry 相关统计。

第六步，增加失败查询接口。

在 `services/api/app.py` 增加：

## 13. 不建议现在做的方向

### 13.1 不建议上真正分布式集群

- 本机资源有限。
- Docker 单机已经足够说明 Kafka/Flink 架构。


### 13.2 不建议接线上付费 ASR

原因：

- 已经明确本地化。
- 线上 API 会削弱“本地部署、可控、低成本”的亮点。
- 课程答辩时网络和额度都有风险。

### 13.3 不建议过度重构前端

原因：

- 当前桌面端和实时端已经能展示主要功能。
- 课设评分更看重系统完整性和实验结果。
- 前端大改容易引入新 bug。

更建议做小增强：

- 增加质量评测入口。
- 增加监控曲线。
- 增加失败片段查看。
- 增加证据包导出按钮。

### 13.4 不建议追求非常低延迟

原因：

- Whisper 类模型天然有推理耗时。
- 本机 GPU 和 ASR 队列是主要瓶颈。
- 课设更重要的是把延迟指标测出来、解释清楚。

更好的目标：

```text
实时链路用于展示流式处理能力。
离线链路用于生成高质量最终字幕。
两条链路的目标不同，所以配置也不同。
```

## 14. 最终建议

如果只做一个增强，做“字幕质量评测闭环”。

如果做两个，增加“一键答辩证据包”。

如果做三个，继续加“实时监控历史曲线”。

如果做四个，再加“失败重试和死信记录”。

这样项目会形成完整闭环：

```text
能输入真实视频
-> 能流式处理
-> 能本地识别
-> 能导出字幕
-> 能量化质量
-> 能量化性能
-> 能追踪失败
-> 能整理答辩材料
```

这就是我认为当前项目最值得继续求精的 v1 方向。
