# StreamSense 本次性能压测实验报告

## 1. 实验目的

本次实验的目标不是验证离线字幕脚本能不能快速生成字幕，而是验证 StreamSense 作为一个多路视频流实时语音分析系统时，能不能做到下面几件事：

1. 能接收真实视频输入，而不是模拟文本或假数据。
2. 能把视频音频按 VAD 动态切成片段。
3. 能通过 Kafka 把音频片段发送给 Flink。
4. 能由 Flink 调度本地 faster-whisper ASR 服务完成转写。
5. 能把转写结果再写回 Kafka，并由 FastAPI 聚合。
6. 能按 `stream_id` 区分不同视频流。
7. 能输出端到端延迟、ASR 耗时、Kafka/Flink 调度耗时、API 聚合耗时、Redis 写入耗时、P50/P95/P99、成功数、失败数、吞吐量和 Kafka 积压。

简单说，原来的项目重点是“能不能生成字幕”，本次实验重点是“这个实时链路能不能被量化、压测和解释”。

## 2. 实验对象

本次实验验证的是完整 Docker 链路：

```text
真实视频 /videos/input.mp4
  -> FFmpeg 按实时速度读取视频
  -> WebRTC VAD 动态切片
  -> Kafka audio-segment
  -> Flink transcription_job.py
  -> faster-whisper ASR
  -> Kafka transcription-result
  -> FastAPI 聚合、句子缓冲、关键词分析、动态热词池
  -> Redis + data/results + API 查询 + 字幕导出
```

这里和 `tools/generate_video_subtitles.py` 的离线字幕生成方式不同。离线字幕脚本通常直接调用 ASR 处理完整视频，目标是尽快生成最终字幕；本实验中的 ingest 使用实时读取方式，更接近摄像头、RTSP 或直播流输入，因此耗时会更长。

## 7. 指标解释

### 7.1 成功片段数

`success_segments` 表示 API 成功接收、聚合并写入的字幕片段数。它不是原始 wav 切片数，而是句子缓冲之后的可展示字幕片段数。

### 7.2 失败片段数

`failed_segments` 表示 Flink 或 ASR 返回错误的片段数量。本次 1 路、2 路、4 路测试中该值都是 0。

### 7.3 端到端延迟

端到端延迟大致对应：

```text
created_at -> result_written_at
```

也就是从 ingest 创建音频片段消息，到 API 完成 Redis/文件写入之间的时间。

### 7.4 ASR 推理耗时

ASR 推理耗时对应：

```text
asr_inference_time_ms
```

它主要反映 faster-whisper 对单个音频片段的识别时间。

### 7.5 Kafka/Flink 调度耗时

Kafka/Flink 调度耗时对应：

```text
kafka_sent_at -> flink_received_at
```

如果这个值很大，通常说明 Kafka 中有积压，或者 Flink/ASR 消费速度跟不上生产速度。

### 7.6 API 聚合耗时

API 聚合耗时对应：

```text
api_received_at -> result_written_at
```

它包括 API 收到转写结果后的句子缓冲、关键词提取、动态热词学习、Redis 写入、JSONL 写入等步骤。

### 7.7 Redis 写入耗时

Redis 写入耗时对应：

```text
redis_write_time_ms
```

本次实验中 Redis 写入基本在 1 到 5 ms 范围内，不是主要瓶颈。

### 7.8 P50 / P95 / P99

P50 表示一半片段延迟低于该值。

P95 表示 95% 的片段延迟低于该值。

P99 表示 99% 的片段延迟低于该值。

实时系统更应该关注 P95/P99，因为用户实际感受到的卡顿往往来自尾部延迟，而不是平均值。

## 9. 单路结果分析

单路结果如下：

```text
成功片段：83
失败片段：0
平均端到端延迟：296086 ms
P50：300823 ms
P95：467350 ms
P99：492275 ms
ASR 平均耗时：1550 ms
Kafka/Flink 平均调度：294156 ms
API 平均聚合：789 ms
Redis 平均写入：2.14 ms
Kafka lag：0
```

这组数据不能直接作为系统正常性能基准，因为它是在 Flink 从 Kafka `earliest` 补消费旧消息时产生的。换句话说，Flink 启动后不是只处理本次新进入的视频流，还把之前 Kafka 中已经存在的旧 `audio-segment` 消息也拿出来处理了。

因此，单路的 Kafka/Flink 调度耗时被历史积压显著放大，导致端到端 P95 达到 467 秒左右。

这组数据的价值在于说明一个现象：

```text
当 Kafka 中存在历史积压时，即使 ASR 单片段推理只需要 1 到 3 秒，端到端延迟也会被排队时间严重拉高。
```

所以单路结果适合写成“积压场景反例”或“系统瓶颈说明”，不适合作为正常实时性能指标。

## 10. 2 路并发结果分析

2 路并发是本次最有代表性的主结果：

```text
成功片段：230
失败片段：0
平均端到端延迟：4084 ms
P50：3815 ms
P95：6805 ms
P99：7485 ms
ASR 平均耗时：1082 ms
Kafka/Flink 平均调度：1398 ms
API 平均聚合：1602 ms
Redis 平均写入：1.78 ms
Kafka lag：0
```

这说明在 2 路并发输入下，系统可以稳定完成：

1. 两路视频流同时进入 ingest。
2. 每一路使用不同 `stream_id`。
3. Kafka 根据 `stream_id` 作为 key 发送消息。
4. Flink 可以消费并调度 ASR。
5. API 可以按流分别查询字幕片段、热词和指标。
6. 最终没有失败片段。
7. Kafka 积压可以清零。

2 路并发时，P95 约为 6.8 秒，P99 约为 7.5 秒。对于单机课程项目原型来说，这个结果是可以接受的。它说明系统已经具备多路实时处理能力，但仍然不是工业级低延迟系统。

从耗时构成看：

| 阶段 | 平均耗时 |
| --- | ---: |
| ASR 推理 | 1082 ms |
| Kafka/Flink 调度 | 1398 ms |
| API 聚合 | 1602 ms |
| Redis 写入 | 1.78 ms |

Redis 很快，几乎不是瓶颈。主要耗时集中在 ASR、Flink 调度和 API 句子缓冲/关键词处理。

## 11. 4 路短冒烟结果分析

4 路结果如下：

```text
成功片段：39

平均端到端延迟：9865 ms
P50：10119 ms
P95：14463 ms

```

这次 4 路是短冒烟测试，不是完整长视频压测。它的主要目的不是得到非常稳定的长期性能数据，而是证明系统可以同时处理 4 个不同 `stream_id` 的视频流。

每路结果：

| stream_id | 成功片段 | 平均端到端延迟 | P95 |
| --- | ---: | ---: | ---: |
| `bench-four-20260428233742-1` | 10 | 8351 ms | 13050 ms |
| `bench-four-20260428233742-2` | 9 | 9370 ms | 14097 ms |
| `bench-four-20260428233742-3` | 10 | 10432 ms | 15065 ms |
| `bench-four-20260428233742-4` | 10 | 11258 ms | 15976 ms |

可以看到，4 路并发时延迟明显比 2 路更高。最明显增长的是 Kafka/Flink 调度耗时：

```text
2 路 Kafka/Flink 平均调度：1398 ms
4 路 Kafka/Flink 平均调度：6711 ms
```

这说明当输入路数增加后，系统开始出现排队。排队不主要发生在 Redis，而是发生在 Kafka/Flink/ASR 调度链路中。

## 12. 为什么 ASR 平均耗时没有明显增加

2 路和 4 路的 ASR 平均耗时都约为 1082 ms。这说明单个音频片段送进 ASR 后，模型推理速度本身比较稳定。

但是端到端延迟从 2 路的 4 秒左右涨到 4 路的 10 秒左右，说明增加的时间主要不是模型“算得更慢”，而是更多片段在进入 ASR 前排队。

当前 ASR 服务中有推理锁，多个请求不会真正同时占满多个 ASR worker。这样做的好处是单机 GPU 更稳定，不容易显存爆掉；坏处是多路并发时会出现等待。

## 13. 为什么 Redis 不是瓶颈

Redis 写入数据：

```text
单路平均：2.14 ms
2 路平均：1.78 ms
4 路平均：2.28 ms
```

这说明 Redis 写入耗时非常小。即使 4 路并发，Redis P95 也只有 5 ms。因此后续优化不应该优先改 Redis，而应该优先看 ASR 并行度、Flink 并行度和 Kafka backlog 策略。

## 14. 为什么离线字幕更快

用户原来使用 `tools/generate_video_subtitles.py` 生成字幕时，一个 10 分钟视频可能只需要两三分钟。这是合理的，因为离线字幕生成通常走的是更短路径：

```text
视频文件 -> ASR /transcribe-media -> 字幕文件
```

本次压测走的是实时流系统路径：

```text
FFmpeg -re 实时读取 -> VAD -> Kafka -> Flink -> ASR -> Kafka -> API -> Redis/JSONL
```

其中 `-re` 会让 FFmpeg 按视频原始速度读取。也就是说，10 分钟视频在实时 ingest 阶段理论上就接近 10 分钟，不会像离线脚本一样全速扫过去。

因此：

```text
离线字幕脚本：适合快速生成最终字幕。
实时压测脚本：适合验证实时系统链路和多路并发能力。
```

两者目标不同，不能直接用运行时间互相比较。

## 15. 本次实验结论

本次实验可以得出下面结论：

1. StreamSense 已经可以完整跑通真实视频流语音分析链路。
2. 系统支持按 `stream_id` 区分多路视频流。
3. 2 路并发结果较稳定，230 个成功片段，失败 0，P95 约 6.8 秒，Kafka 最终无积压。
4. 4 路短冒烟可以跑通，失败 0，但 P95 上升到约 14.5 秒，说明多路下出现明显排队。
5. 单路异常高延迟主要由 Kafka 历史积压导致，不代表干净环境下的正常性能。
6. ASR 单片段推理大约 1 秒左右，比较稳定。
7. Redis 写入只有几毫秒，不是瓶颈。
8. 当前主要瓶颈是 Kafka/Flink/ASR 调度链路，尤其是 ASR 单机排队。

一句话总结：

```text
对课程项目来说，本次结果是好的；它已经不是单纯字幕工具，而是可量化、可压测、可解释的多路视频流实时语音分析系统原型。
```

## 17. 后续优化建议

后续如果想让数据更漂亮、更接近工业系统，可以按下面顺序做：

1. 压测前清理 Kafka topic 或使用新的 topic，避免历史积压影响结果。
2. 把 Flink starting offset 从 `earliest` 改成可配置，压测时使用 `latest`。
3. 增加 ASR worker 池，例如启动 2 到 4 个 ASR 服务实例。
4. 让 Flink 根据 `stream_id` 分区后并行调用不同 ASR worker。
5. 把 API 的句子缓冲、关键词提取和热词更新做成更轻的异步任务。
6. 单独记录 ingest 实时读取耗时、ASR 队列等待耗时和模型实际推理耗时，进一步拆开瓶颈。
7. 做一组干净环境下的完整 1 路、2 路、4 路全量视频测试，生成最终可放进论文的表格。
