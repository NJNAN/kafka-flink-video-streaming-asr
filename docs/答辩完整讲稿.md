# StreamSense 课程设计答辩完整讲稿

这份讲稿适合 8 到 12 分钟答辩。不要逐字机械背诵，建议按“问题背景 -> 系统架构 -> 核心实现 -> 求精增强 -> 实验结果 -> 演示 -> 总结”的顺序讲。答辩时要把重点放在“我不是只调了一个语音识别模型，而是做了一条可运行、可观测、可评测、可容错的大数据流式处理链路”。

## 1. 开场

老师好，我的课程设计题目是：

```text
基于 Kafka-Flink 的视频流语音转写与关键词分析系统
```

我做这个项目的目标不是简单把一个视频丢给 Whisper，然后生成字幕文件，而是把视频流、实时音频流中的语音内容，接入到 Kafka 和 Flink 组成的流式处理链路里，完成实时转写、关键词分析、指标监控、字幕导出和实验评测。

简单来说，这个系统做了三件事：

1. 把真实视频、摄像头、麦克风或直播流里的声音切成连续音频片段。
2. 通过 Kafka 和 Flink 做流式调度，由本地 faster-whisper ASR 服务完成语音识别。
3. 把转写结果沉淀到 Redis、JSONL、SQLite 和前端 Dashboard 中，支持实时展示、字幕导出、质量评测和压测分析。

所以我的项目不是一个单机字幕脚本，而是一个面向流式场景的视频语音分析系统。

## 2. 项目解决的问题

我这个课题面向的实际问题是：很多课程视频、会议录屏、直播和摄像头场景中，语音内容很难被结构化利用。如果只是保存视频文件，后续想搜索关键词、查看字幕、统计内容或做实时展示都比较麻烦。

所以我把问题拆成了几个工程目标：

- 第一，能处理真实视频和真实音频，而不是模拟文本数据。
- 第二，能把连续音频流拆成可处理的语音片段。
- 第三，能用 Kafka 解耦数据接入和下游处理，避免上游和 ASR 服务强耦合。
- 第四，能用 Flink 进行持续消费和流式调度。
- 第五，能调用本地 ASR 模型完成转写，不依赖在线付费 API。
- 第六，能把结果实时展示、导出、评测和统计。

这也是我认为这个项目能体现“大数据课程设计”的地方：它的核心不是语音识别模型本身，而是围绕语音识别构建了一条流式数据处理链路。

## 3. 系统总体架构

系统可以分成五层。

第一层是数据接入层。它负责从视频文件、摄像头、麦克风或实时音频流中读取声音，然后通过 FFmpeg 抽取音频，再使用 VAD 进行语音活动检测，把连续音频拆成一个个音频片段。

第二层是消息队列层。音频片段不会直接传给 ASR，而是先写入 Kafka 的 `audio-segment` topic。这样做的好处是上游接入和下游处理解耦，如果 ASR 处理速度变慢，Kafka 可以暂时承接积压，系统不会马上崩掉。

第三层是流处理层。Flink 作业会持续消费 Kafka 中的音频片段，并且给每个片段补充链路时间戳，然后调用 ASR 服务完成转写。Flink 在这里承担的是流式任务调度、失败处理和多路流处理。

第四层是服务和存储层。这里包括本地 faster-whisper ASR 服务、FastAPI 服务、Redis 缓存、JSONL 结果文件和 SQLite 结构化结果库。Redis 负责实时展示，JSONL 负责保留原始运行结果，SQLite 负责后续查询和统计。

第五层是展示和工具层。包括 Web Dashboard、离线字幕桌面端、大数据实时字幕桌面端、质量评测脚本、压测脚本、冒烟测试脚本和答辩材料打包脚本。

整体链路可以概括成：

```text
真实视频 / 摄像头 / 麦克风
-> FFmpeg 抽音频
-> VAD 切分语音片段
-> Kafka audio-segment
-> Flink 转写调度
-> 本地 faster-whisper ASR
-> Kafka transcription-result
-> FastAPI 聚合结果
-> Redis / JSONL / SQLite
-> Dashboard / 桌面端 / 字幕导出 / 实验报告
```

## 4. 两条使用链路

项目里我保留了两条链路，它们服务于不同目标。

第一条是离线字幕链路。它主要用于处理一个已经存在的视频文件，目标是生成质量较好的最终字幕文件。链路是：

```text
desktop-ui
-> tools/generate_video_subtitles.py
-> ASR 服务
-> SRT / VTT / TXT / JSON / REPORT
```

这条链路适合答辩时展示“这个系统确实能把视频变成可用字幕”。

第二条是大数据实时字幕链路。它走完整的 Kafka + Flink + ASR + API 链路，目标是展示实时流式处理能力。链路是：

```text
desktop-ui-live
-> live-ingest
-> Kafka
-> Flink
-> ASR
-> Kafka
-> FastAPI
-> Redis + Dashboard
```

这条链路更能体现课程设计中的大数据技术栈。

## 5. 核心模块实现

### 5.1 音频接入和 VAD 切片

系统接入视频后，不是把整段视频一次性送给模型，而是先用 FFmpeg 抽取音频，再进行语音活动检测。VAD 的作用是判断哪些区间有人声，哪些区间是静音或背景噪声。

这样做有三个好处：

- 减少无效静音片段进入 ASR，降低计算浪费。
- 让长视频变成连续小片段，更适合流式处理。
- 方便后续统计每个片段的端到端延迟。

这一步体现的是数据接入和预处理能力。

### 5.2 Kafka 解耦

Kafka 在系统中不是摆设。它承担了两个核心 topic：

```text
audio-segment
transcription-result
```

音频片段先进入 `audio-segment`，Flink 消费后调用 ASR，再把识别结果写入 `transcription-result`。API 服务再消费识别结果，进行聚合、缓存和展示。

我这样设计的原因是：音频接入、ASR 推理、API 展示三者的速度并不一致。如果直接同步调用，任何一个环节卡住都会影响整条链路。Kafka 可以把这些环节拆开，让系统更接近真实的大数据流处理架构。

### 5.3 Flink 流式转写调度

Flink 的主要任务是持续消费音频片段，调用 ASR 服务，并把结果重新写回 Kafka。它还会记录链路时间戳，例如：

```text
created_at
kafka_sent_at
flink_received_at
asr_start_at
asr_end_at
api_received_at
result_written_at
```

有了这些时间戳，我就可以计算：

- 端到端延迟
- Kafka/Flink 调度耗时
- ASR 推理耗时
- API 聚合耗时
- Redis 写入耗时
- P50、P95、P99 尾部延迟

这也是后面性能压测和监控曲线的基础。

### 5.4 本地 ASR 服务

ASR 服务使用 faster-whisper。它的优势是可以在本地部署，支持 CPU 或 CUDA，也支持不同模型大小和计算精度，比如：

```text
ASR_MODEL=small / medium / large-v3
ASR_DEVICE=cpu / cuda
ASR_COMPUTE_TYPE=int8 / int8_float16 / float16
```

这样我可以根据机器性能在准确率和实时性之间做权衡。比如 GPU 条件好时可以用 `large-v3 + float16`，显存不够时可以切到 `medium` 或 `small`。

### 5.5 API、缓存和展示

FastAPI 服务负责消费转写结果，并完成几件事：

- 聚合同一路 `stream_id` 的字幕片段。
- 保存实时字幕和关键词到 Redis。
- 写入 JSONL 运行日志。
- 写入 SQLite 结构化结果库。
- 提供 Dashboard 和 REST API。
- 提供字幕导出接口。

核心接口包括：

```text
/api/metrics
/api/metrics/history
/api/streams
/api/streams/{stream_id}/segments
/api/streams/{stream_id}/export
/api/failed-segments
/api/database/summary
```

这些接口让系统不是“黑盒运行”，而是可以被观察、查询和解释。

## 6. 项目求精增强

原始版本能跑通 Kafka-Flink-ASR 主链路，但如果答辩时老师追问“准确率怎么证明、失败怎么办、结果怎么统计、系统是否可观测”，只靠能跑是不够的。所以我后面做了求精增强。

### 6.1 字幕质量评测闭环

我新增了字幕质量评测工具：

```text
tools/evaluate_subtitles.py
docs/字幕质量评测说明.md
```

它支持用人工参考文本或参考字幕作为标准答案，对系统生成的字幕计算：

- CER，字符错误率，适合中文转写评价。
- WER，词错误率，适合英文或带空格文本。
- 关键词命中率，用来评价 Kafka、Flink、Whisper、Redis 等领域词是否识别准确。
- 字幕条目数量、字符数、编辑距离等辅助指标。

这部分解决的是“字幕准不准怎么证明”的问题。以前只能人工看，现在可以给出量化指标。

### 6.2 实时监控历史曲线

我新增了指标历史采样和前端曲线：

```text
/api/metrics/history
data/results/metrics_history.jsonl
data/results/streamsense.db
Dashboard 历史曲线
```

监控指标包括：

- 平均端到端延迟
- P95 延迟
- 吞吐量
- 成功片段数
- 失败片段数
- ASR 平均耗时
- retry_count

这样答辩时可以说明：系统不仅能处理数据，还能持续观测自己的运行状态。

### 6.3 失败重试和死信记录

实时系统中 ASR 可能超时，服务也可能短暂不可用。所以我在 Flink 转写逻辑里加了失败重试机制。

处理逻辑是：

```text
调用 ASR
-> 如果失败，按配置重试
-> 重试仍失败，写入 transcription-failed
-> API 记录 failed_segments
-> Dashboard 和接口可查询失败片段
```

对应的配置包括：

```text
FAILED_TOPIC=transcription-failed
ASR_RETRY_TIMES=2
ASR_RETRY_BACKOFF_MS=800
```

这部分体现的是工程容错能力，不是只写一条正常路径。

### 6.4 SQLite 结构化结果库

原来系统主要依赖 Redis 和 JSONL。Redis 适合实时缓存，JSONL 适合保留原始日志，但如果要统计实验结果、查询某一路 stream 的平均延迟或失败数，就不够方便。

所以我增加了 SQLite 结果库：

```text
services/api/storage.py
tools/query_results.py
data/results/streamsense.db
```

数据库中保存：

- runs
- segments
- failed_segments
- metrics_samples
- evaluations

这样系统的结果就从“运行日志”提升成了“可查询的实验数据”。

### 6.5 领域 Profile

我还增加了领域 Profile 管理：

```text
config/profiles/
```

比如：

```text
bigdata_keywords.txt
bigdata_corrections.txt
course_keywords.txt
course_corrections.txt
meeting_keywords.txt
meeting_corrections.txt
```

不同场景可以切换不同热词和纠错表。比如大数据课设场景会强化 Kafka、Flink、Redis、Whisper、checkpoint 等词；会议场景可以强化会议纪要、人名或项目名。

这让系统更像一个可配置的产品，而不是写死在一个视频上的脚本。

### 6.6 自动化冒烟测试和答辩证据包

为了答辩前快速确认系统是否健康，我新增了：

```text
tools/smoke_check.py
```

它会检查：

- API 是否正常
- ASR 是否正常
- Flink Job 是否 RUNNING
- Docker 服务是否启动
- Kafka topic 是否存在
- 指标接口是否返回

同时我还新增了：

```text
tools/make_defense_package.py
```

它可以一键整理 README、文档、字幕样例、评测报告、压测报告和系统状态，形成答辩证据包。

这部分体现的是工程交付意识：不是只写代码，还要能验证、能提交、能展示。

## 7. 性能实验与结果

我做过 1 路、2 路、4 路的完整链路压测。测试对象不是离线字幕脚本，而是 Docker 中的完整实时链路：

```text
FFmpeg 实时读视频
-> VAD 切片
-> Kafka
-> Flink
-> ASR
-> Kafka
-> API
-> Redis / JSONL / 指标
```

核心结果如下：

| 测试场景 | 成功片段 | 失败片段 | 平均端到端延迟 | P95 | 结论 |
| --- | ---: | ---: | ---: | ---: | --- |
| 单路 | 83 | 0 | 296086 ms | 467350 ms | 受历史 Kafka 积压影响，不作为干净基准 |
| 2 路并发 | 230 | 0 | 4084 ms | 6805 ms | 最适合作为本次报告主结果 |
| 4 路短冒烟 | 39 | 0 | 9865 ms | 14463 ms | 证明 4 路可跑，但排队延迟上升 |

这里我重点解释 2 路结果。2 路并发场景下，系统成功处理 230 个片段，失败片段为 0，平均端到端延迟约 4.1 秒，P95 延迟约 6.8 秒。

从指标拆解看，主要耗时来自 ASR 推理和多路并发下的调度排队，Redis 写入通常只有几毫秒，不是瓶颈。这说明系统瓶颈主要在语音识别模型推理和并发调度上，而不是 API 或 Redis。

4 路短冒烟说明系统可以同时处理多路 stream，但是随着并发路数增加，排队延迟明显上升。这个现象符合预期，因为本地 ASR 模型是计算密集型服务。

## 8. 技术亮点总结

如果要体现技术力，我认为这个项目的亮点主要有八个。

第一，真实数据流。系统处理的是真实视频、摄像头、麦克风或直播流，不是模拟文本。

第二，完整流式架构。数据从接入、Kafka、Flink、ASR、Kafka、API 到前端展示形成完整闭环。

第三，本地模型服务化。ASR 不是直接嵌在脚本里，而是封装成 HTTP 服务，方便被 Flink 调用和单独扩缩。

第四，多路流隔离。系统通过 `stream_id` 区分不同输入流，支持多路并发压测和单路导出。

第五，细粒度指标。系统保留多个链路时间戳，可以拆解端到端延迟、ASR 耗时、Kafka/Flink 调度耗时、API 聚合耗时和 Redis 写入耗时。

第六，失败容错。Flink 调用 ASR 时支持重试，失败后进入死信 topic，并可通过 API 查看失败片段。

第七，结果可沉淀。Redis 负责实时，JSONL 负责原始日志，SQLite 负责结构化查询，三者各司其职。

第八，实验闭环。项目有压测、质量评测、冒烟测试、模型对比和答辩材料整理，不只是功能演示。

答辩时可以把这八点作为“高技术力”的集中表达。

## 10. 常见问题回答

### 10.1 老师问：你的项目为什么算大数据项目？

可以这样答：

这个项目的大数据特征主要体现在流式处理架构上。视频和音频不是一次性脚本处理，而是被切成连续音频片段进入 Kafka，由 Flink 持续消费和调度，再调用 ASR 服务完成转写，最后通过 API 聚合和展示。Kafka 负责消息缓冲和上下游解耦，Flink 负责流处理调度和失败处理，系统还支持多路 stream、压测、延迟统计和历史指标，所以它不是普通的单机字幕生成脚本。

### 10.2 老师问：为什么不用一个 Python 脚本直接转写？

可以这样答：

直接 Python 脚本确实可以生成字幕，但它不能体现实时流处理、消息队列解耦、多路并发、失败重试、端到端延迟监控和结构化结果沉淀。本项目的目标不是只得到一份字幕，而是构建一条可运行、可观测、可评测、可扩展的语音流处理链路。

### 10.3 老师问：Kafka 在这里有什么必要？

可以这样答：

Kafka 主要解决解耦和削峰问题。音频接入速度、ASR 推理速度和前端消费速度并不一致，如果直接同步调用，ASR 一慢就会拖垮前面的视频接入。Kafka 把音频片段先缓存起来，让 Flink 按能力消费，也方便后续扩展多消费者、失败重放和多路流统计。

### 10.4 老师问：Flink 具体做了什么？

可以这样答：

Flink 负责消费 Kafka 中的音频片段，给每个片段补充链路时间戳，调用 ASR 服务，把结果写回 Kafka，并处理失败重试。它在系统里承担的是流式调度层，而不是简单函数调用。通过 Flink，我可以把每个音频片段看成流上的一条事件，并统计它在整个链路中的处理耗时。

### 10.5 老师问：准确率怎么证明？

可以这样答：

我新增了字幕质量评测模块。可以把人工整理的参考文本作为标准答案，对系统生成的字幕计算 CER、WER 和关键词命中率。CER 更适合中文语音转写评价，关键词命中率能反映 Kafka、Flink、Whisper、Redis 等领域词有没有识别出来。这样识别效果不是只靠人工主观判断，而是可以量化。

### 10.6 老师问：系统失败怎么办？

可以这样答：

Flink 调用 ASR 时支持重试，重试次数和退避时间可以配置。如果重试后仍然失败，片段会进入 `transcription-failed` 死信 topic，同时 API 会记录失败片段，Dashboard 和 `/api/failed-segments` 可以查询。这样系统不会因为一个片段失败就中断整条流。

### 10.7 老师问：你的瓶颈在哪里？

可以这样答：

从压测指标看，主要瓶颈在 ASR 推理和并发增加后的调度排队。Redis 写入一般只有几毫秒，不是瓶颈。2 路并发时平均延迟约 4 秒，4 路短冒烟时 P95 延迟明显上升，说明本地 ASR 是计算密集型环节。如果后续优化，可以考虑 ASR 服务多副本、GPU 更大模型并行、Flink 并行度调优和片段长度调优。

### 10.8 老师问：你后续还能怎么优化？

可以这样答：

后续可以从四个方向优化：

- 模型层：比较 small、medium、large-v3 在准确率和延迟上的权衡。
- 架构层：给 ASR 服务做多副本，把 Flink 并行度和 ASR 副本数配合起来。
- 质量层：增加更多人工参考文本，形成更完整的 CER/WER 评测集。
- 产品层：增强 Dashboard，把失败重放、热词确认和字幕人工修订做成界面操作。

## 11. 结尾总结

所以它不是一个只会生成字幕的小工具，而是一个围绕视频语音数据构建的流式大数据处理系统。它既能跑真实数据，也能解释性能瓶颈，还能通过指标和报告证明处理效果。
