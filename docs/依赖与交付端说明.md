# 依赖与交付端说明

本文说明 StreamSense 的依赖文件为什么按模块拆分，以及四个前端/端侧入口各自承担什么职责。

## 1. 依赖文件分层

StreamSense 不是单脚本项目，而是由 Kafka、Flink、ASR、API、接入服务、桌面端和移动端组成的多服务工程。依赖按模块拆分，可以避免把 PyFlink、faster-whisper、CUDA 推理依赖、Electron 和 Android 构建工具全部安装到同一个环境里。

| 文件 | 模块 | 主要依赖 | 用途 |
| --- | --- | --- | --- |
| `requirements.txt` | 根目录工具与轻量测试 | 当前只使用 Python 标准库 | 运行 `tools/smoke_check.py`、`tools/evaluate_subtitles.py`、`python -m unittest` |
| `services/api/requirements.txt` | API + Dashboard 聚合服务 | FastAPI、aiokafka、Redis、jieba、pydantic | 消费转写结果、关键词分析、指标接口、Dashboard |
| `services/asr/requirements.txt` | ASR 识别服务 | faster-whisper、OpenCC、FastAPI、aiokafka | 本地 Whisper 推理、动态热词接收、繁简转换 |
| `services/ingest/requirements.txt` | 视频接入服务 | kafka-python、webrtcvad-wheels | FFmpeg 抽音频、VAD 切片、写入 Kafka |
| `flink/requirements.txt` | PyFlink 作业 | apache-flink、requests | 消费 `audio-segment`、调度 ASR、写回 `transcription-result` |
| `desktop-ui-live/live-ingest/requirements.txt` | 实时音频接入 | FastAPI、kafka-python、python-multipart | 接收桌面端/移动端音频分片，转换后写入 Kafka |
| `subtitle-agent/requirements.txt` | 可选字幕增强 | textual | 终端交互式字幕审校与增强 |
| `desktop-ui/package.json` | 离线桌面端 | Electron、React、Vite、TypeScript | 本地视频字幕工作台 |
| `desktop-ui-live/package.json` | 实时桌面端 | Electron、React、Vite、TypeScript | 摄像头/麦克风实时字幕采集端 |
| `meeting-assistant-tablet/package.json` | 手机/平板端 | React、Vite、Capacitor Android | MeetFlow 会议纪要 App |

## 2. 安装策略

常规后端运行优先使用 Docker Compose：

```powershell
docker compose up -d --build
```

本地只跑工具和轻量测试时使用根目录依赖入口：

```powershell
python -m pip install -r requirements.txt
python -m unittest discover -s tests -v
```

某个服务需要脱离 Docker 调试时，再安装对应模块依赖：

```powershell
python -m pip install -r services/api/requirements.txt
python -m pip install -r services/asr/requirements.txt
python -m pip install -r desktop-ui-live/live-ingest/requirements.txt
```

前端入口使用各自目录的 `package.json`：

```powershell
cd desktop-ui
npm install
npm run electron:dev
```

## 3. 四个交付端

同一条 `音频 -> Kafka -> Flink -> ASR -> API` 主链路被封装为四个入口。

| 交付端 | 目录 | 输入 | 核心能力 | 输出 |
| --- | --- | --- | --- | --- |
| Web Dashboard | `services/api/static/` | 后端聚合结果 | 实时字幕、关键词、延迟、吞吐、失败片段、历史曲线 | 浏览器观测页 |
| 离线 Electron 工作台 | `desktop-ui/` | 本地视频文件 | 生成字幕、展示质量报告、字幕时间轴、字幕编辑、多格式导出 | SRT/VTT/TXT/JSON/ZIP |
| 实时 Electron 采集端 | `desktop-ui-live/` | 摄像头/麦克风 | 启停实时服务、分片上传、Live Ingest 写 Kafka、轮询 `desktop-live` 字幕 | 实时字幕界面 |
| MeetFlow 手机/平板 App | `meeting-assistant-tablet/` | 移动端麦克风 | 1.8 秒音频切片、会议热词、实时文字、摘要、待办、原文摘录 | PWA / Android APK |

## 4. 验证入口

| 目标 | 命令或入口 | 预期结果 |
| --- | --- | --- |
| 后端核心服务 | `python tools/smoke_check.py` | API、ASR、Flink、Docker、Kafka Topic、指标接口通过 |
| Dashboard | `http://localhost:8000` | 可以看到实时字幕、关键词、延迟和失败片段 |
| 离线桌面端 | `cd desktop-ui && npm run build` | React/TypeScript/Vite 构建通过 |
| 实时桌面端 | `cd desktop-ui-live && npm run build` | 实时采集端前端构建通过 |
| 移动端 PWA | `cd meeting-assistant-tablet && npm run build` | MeetFlow Web 构建通过 |
| Android APK | `cd meeting-assistant-tablet && npx cap sync android && .\android\gradlew.bat -p android assembleDebug` | 生成 debug APK |

## 5. 为什么不做一个巨大的 requirements.txt

一个全量 `requirements.txt` 会同时拉入 PyFlink、faster-whisper、CTranslate2、Web 服务依赖和接入服务依赖，安装慢、冲突概率高，也不符合本项目的容器化边界。当前拆分方式更清晰：

- 服务运行靠 Docker 镜像和各自 `requirements.txt`。
- 本地脚本和测试靠根目录 `requirements.txt`。
- 前端和移动端靠各自 `package.json`。
- Android 构建靠 `meeting-assistant-tablet/android/` 下的 Gradle 工程。
